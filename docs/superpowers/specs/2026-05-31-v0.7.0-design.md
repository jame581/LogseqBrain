# LogseqBrain v0.7.0 — Design

**Status:** Draft, pending implementation
**Date:** 2026-05-31
**Target version:** 0.7.0 (from 0.6.0)
**Release shape:** Single bundled minor version (four items, all markdown-skill edits)

## Goals

1. **Durable config location** — survive `/reload-plugins` and version bumps so graph-path resolution no longer breaks between sessions.
2. **Logseq normalization tolerance** — a shared reference that documents Logseq's parse-time normalization and the read-before-edit rules that survive it.
3. **Re-add `HH:mm` time prefix** to journey-log activity bullets — reverse the v0.6.0 "no prefix" call that dogfooding disproved.
4. **Graph analytics ("brain stats")** — an opt-in analytics mode on `brain-status`, promoted from the Future list.

## Out of scope

- Storage abstraction layer / Logseq DB plugin API / headless CLI sync — still deferred until Logseq's direction stabilizes (unchanged from v0.6.0).
- A dedicated 5th skill — brain-stats rides on `brain-status`; skill count stays at 4.
- Conflict resolution for Logseq Sync — remains a Future item.
- Automated tests — the project remains validation-by-hand against a scratch graph (per CLAUDE.md).

## Architecture overview

No structural change to the four-skill layout. v0.7.0 adds one shared reference (`skills/_shared/logseq-format.md`), rewrites the resolution logic in `skills/_shared/path-resolution.md`, and extends `brain-status/SKILL.md` with an analytics mode. The `.brain-config.json`-at-cache-root contract is replaced by a user-config-dir contract.

## Section 1 — Durable config location

### Problem

`.brain-config.json` lives at the cache plugin root. That directory is wiped on `/reload-plugins` and on every plugin version bump, so the persisted graph path is lost and the user is re-prompted every session — or worse, resolution silently fails.

### Resolution order (Claude Code / Copilot CLI / Gemini CLI hosts)

The Cowork branch (`request_cowork_directory`) is unchanged. For the other hosts, `path-resolution.md` resolves the graph path in this order:

1. **`LOGSEQ_BRAIN_PATH` env var** — highest precedence for the graph path. If set and non-empty, use it; don't read or write the config file *for path resolution*. (This scoping is path-only: non-path settings such as the `journeyLog` toggle are still read from the config file by `journey-log.md`, even when the env var supplied the path.)
2. **User config file** — read `graphPath` from:
   - Windows: `%APPDATA%\logseq-brain\config.json`
   - mac/Linux: `$XDG_CONFIG_HOME/logseq-brain/config.json`, falling back to `~/.config/logseq-brain/config.json`
3. **Ask the user**, then **persist** the answer to the user config file from step 2 (creating the directory if needed).

### Config file shape

JSON, a superset of the old `.brain-config.json` so the existing toggle survives:

```json
{
  "graphPath": "C:\\Users\\me\\ClaudeBrain",
  "journeyLog": true
}
```

The `journeyLog` toggle (default `true`) moves from `.brain-config.json` into this file. Skills that read the toggle (`journey-log.md` callers) read it from the resolved config file.

### One-time migration

When resolving: if the env var is unset **and** the new user config file is absent, check for a legacy `.brain-config.json` at the cache plugin root. If found, read it and write its values to the new user config file (best-effort, silent — failure just falls through to "ask the user"). This makes the upgrade seamless for the existing dogfood graph without a manual step.

### Touches

`skills/_shared/path-resolution.md` (rewrite), `CLAUDE.md` (path-resolution description), `CONTRIBUTING.md` (validation checklist), `README.md` (install/setup instructions), and every `.brain-config.json` reference across the four `SKILL.md` files and references.

## Section 2 — `skills/_shared/logseq-format.md`

### Problem

When Logseq parses pages/journals we wrote, it normalizes them: it drops the `- ` prefix from `##` heading bullets, converts leading-space indents to tabs, and strips empty section headings (e.g. an empty `## Sessions`). Surgical Edits already cope via Read-before-Edit, but the algorithm prose in `journey-log.md` and `brain-save/SKILL.md` assumes our *written* format survives, which is misleading and fragile.

### Design

A single shared reference, `skills/_shared/logseq-format.md`, documenting:

- **The normalizations** Logseq applies on parse (heading-bullet stripping, space→tab indent conversion, empty-heading removal).
- **Survival rules:**
  - Always Read the target region *immediately before* Edit — never edit against remembered/written content.
  - Match Edit `old_string` against the **normalized** form (e.g. `## Activity`, not `- ## Activity`).
  - Anchor insertions on **heading text**, not on a bullet prefix or indentation.
  - Treat a missing-but-expected empty heading as "Logseq removed it" — recreate it rather than assuming corruption.

`brain-save/SKILL.md` and `journey-log.md` each gain a one-line pointer to this reference where they currently assume written-format survival.

## Section 3 — Re-add `HH:mm` time prefix to journey-log bullets

### Rationale

v0.6.0 dropped the time prefix on the theory that Logseq auto-assigns `created-at::` block metadata for inline display. Dogfooding (2026-05-02) showed Logseq does **not** render those timestamps visibly, so the activity trail lost its quick-scan chronological signal.

### Design

Journey-log activity bullets become:

```
- HH:mm <activity>
```

in local 24-hour time (e.g. `- 14:32 Loaded project ClaudeBrain (brief)`). The time is derived from the system clock at write time.

### Touches

`skills/_shared/journey-log.md` (bullet format + examples), the activity-line examples in all four `SKILL.md`, and the events table in `docs/superpowers/specs/2026-05-01-v0.6.0-design.md` (already marked superseded — update to reference this spec).

## Section 4 — Graph analytics ("brain stats") mode on `brain-status`

### Design

Extend `brain-status` rather than add a skill. New trigger phrases — "brain stats", "graph analytics", "activity over time" — are added to the `brain-status` frontmatter `description`, scoped so the plain "show projects / dashboard" triggers still route to the normal dashboard and the analytics phrases route to the stats output.

### What it reports

Scanning project pages + `pages/Decisions.md` + recent journals (using `section-locator` for targeted reads to stay token-frugal), it emits a compact summary:

- **Projects:** total count (and active vs. stale, reusing `staleness.md`).
- **Decisions:** total count, broken down by `status::` (accepted / superseded / etc.).
- **Sessions:** total session-log count across project pages.
- **Activity:** count of journey-log activity bullets in a recent window (last 30 days), as the trend signal. All-time totals for projects/decisions/sessions; recent-window figure for activity.

Output is a single compact bullet block, consistent with the existing dashboard style. No new pages are written — this is a read-only view.

### Touches

`skills/brain-status/SKILL.md` (description triggers + analytics mode branch). May add `skills/brain-status/references/stats.md` if the analytics logic is large enough to warrant progressive disclosure (decided during implementation).
